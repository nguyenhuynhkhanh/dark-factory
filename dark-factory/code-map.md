- **spec-agent.md**: Phase 4a classifies spec complexity into Tier 1/2/3 using a signal table, writes `Architect Review Tier` into spec. This field is the single source of truth for implementation-agent's architect spawn decision.
- **test-agent.md**: outputs `dark-factory/results/{feature}/run-{timestamp}.md` with structured metadata (`flakyE2E` boolean, `devServerSource`). Honors backend-only exclusion (`UI Layer == none`).
- **bin/cli.js**: exports 4 functions (`cmdInit`, `cmdUpdate`, `cmdInitHooks`, `cmdHelp`). Public API via `npx dark-factory {init|update|help}`.

## Cross-Cutting Concerns

- **Information barriers**: Every agent declares explicit NEVER-read constraints. Holdout scenarios hidden from code-agent. Public scenarios hidden from test-agent. Tests/scenarios hidden from architect-agent. Enforced by 50+ test assertions.
- **Code map freshness protocol**: df-intake, df-debug, and df-orchestrate all share the same pre-pipeline hook pattern: read code-map.md header -> compare git hash -> spawn codemap-agent if stale.
- **Slim file ecosystem**: Three agents/skills produce slim files (onboard-agent writes profile-slim, codemap-agent writes code-map-slim, df-cleanup regenerates both). One agent consumes them (architect-agent for Tier 1/2). All share the invariant: slim file `Git hash:` must match the full file.
- **Plugin mirroring**: all agents, skills, rules, and templates must exist in both `.claude/` and `plugins/dark-factory/`. Contracts test enforces exact match. Slim template now included.
- **Worktree isolation**: df-intake, df-debug, df-orchestrate, and implementation-agent (multi-track code-agents) all spawn agents with `isolation: "worktree"` for parallel, conflict-free execution. Wave-based dependency resolution in df-orchestrate.
- **Template-driven output**: spec-agent, debug-agent, and onboard-agent all read structured templates from `dark-factory/templates/` rather than embedding output format inline. `spec-template.md` now includes `Architect Review Tier` field.
- **Pre-flight test gate**: implementation-agent runs project test suite before architect review. Bypassable via `--skip-tests` (logged in manifest as `testGateSkipped: true`).
- **Tiered architect review**: spec-agent classifies tier (Tier 1/2/3) at spec-write time. implementation-agent reads tier at spawn time. architect-agent uses tier to select context loading strategy and round budget. Self-escalation is permitted upward only. Strictest-wins for domain disagreement.
- **Round summarization**: architect-agent writes compact round summaries (max 400 words) after each review round. Next round begins by reading prior summary. Graceful fallback if summary is missing (treat as round 1).
- **UI Layer / E2E detection (Phase 2.5)**: onboard-agent uses an explicit allowlist for frontend frameworks and E2E frameworks. Records 4 rows in profile Tech Stack table consumed by test-agent for backend-only exclusion.
- **Flaky E2E routing**: test-agent emits `flakyE2E: true` boolean in result metadata. implementation-agent uses this AS THE SINGLE AUTHORITATIVE SIGNAL — routes flaky scenarios to spec-agent in bugfix mode, routes clean failures to code-agent re-run.
- **Dev server lifecycle (test-agent)**: 3-tier fail-soft cascade: (1) Playwright config webServer, (2) project-profile `Dev Server` command, (3) `npm run dev` fallback. Always cleans up via process group kill.
- **Git hook installation (Phase 7.5)**: onboard-agent detects existing infrastructure (Husky, Lefthook, simple-git-hooks) and installs into the appropriate location. Idempotent — skips if marker already present.
