## Architecture
### Structure
Dark Factory is a prompt engineering framework, not a traditional application. It consists of:
1. **Agent definitions** (`.claude/agents/*.md`) — markdown files with YAML frontmatter that define agent behavior, access rules, and processes
2. **Skill definitions** (`.claude/skills/*/SKILL.md`) — slash command handlers that orchestrate agent spawning
3. **An init script** (`scripts/init-dark-factory.js`) — scaffolds the framework into any target project
4. **A test suite** (`tests/dark-factory-setup.test.js`) — verifies structural integrity of all agents, skills, barriers, and the scaffold
5. **Runtime directories** (`dark-factory/`) — where specs, scenarios, results, and archives live during use

There is no "running application." The framework operates entirely through Claude Code's agent and skill system. The init script is the only executable code.

### Patterns to Follow
- **Agent definitions**: Markdown with YAML frontmatter (`name`, `description`, `tools`). Body contains role description, process steps, templates, and constraints. Example: `.claude/agents/spec-agent.md`
- **Skill definitions**: Markdown with YAML frontmatter (`name`, `description`). Body contains trigger syntax, process steps, and rules. Example: `.claude/skills/df-intake/SKILL.md`
- **Init script**: All agent/skill content is generated inline as template literal strings in `scripts/init-dark-factory.js`. There are no external template files. Content is written via `fs.writeFileSync`.
- **Information barriers**: Every agent explicitly declares what it CANNOT access using "NEVER read/modify" constraint blocks. These are tested.
- **Frontmatter naming**: The `name` field in frontmatter MUST match the filename (without `.md`) for agents and the directory name for skills. Tests enforce this.

### Shared Abstractions
- `parseFrontmatter()` in `tests/dark-factory-setup.test.js` — parses YAML frontmatter from agent/skill files (test-only utility, not shared with production code)
- `detectProjectType()` in `scripts/init-dark-factory.js` — auto-detects nestjs/node/generic project type
- `getClaudeMdSection()` in `scripts/init-dark-factory.js` — generates the CLAUDE.md Dark Factory documentation block

## Structural Notes

- **All agent content lives inside the init script.** Agent and skill markdown is generated by template literal functions in `scripts/init-dark-factory.js` (lines 83-1021). The `.claude/agents/` and `.claude/skills/` files in this repo are the "source of truth" for the Dark Factory project itself, but for target projects they are generated by the init script. Changes to agent behavior must be made in TWO places: the actual `.md` file in this repo AND the corresponding generator function in `init-dark-factory.js`.

- **No package.json exists.** The project has zero npm dependencies. The README references `npx dark-factory init` but there is no npm package configuration to support this. Currently, the only installation path is `node scripts/init-dark-factory.js --dir <path>`.

- **Project type detection affects agent content.** The init script generates slightly different agent prompts for `nestjs`, `node`, and `generic` project types. This branching only affects `spec-agent` (codebase research instructions) and `code-agent` (implementation patterns section). The `architect-agent`, `test-agent`, `debug-agent`, `onboard-agent`, and `promote-agent` are identical across project types.

- **CLAUDE.md in this repo has a dual role.** It both configures Claude Code for working on the Dark Factory project itself AND serves as the template for what gets appended to target projects' CLAUDE.md files. The `getClaudeMdSection()` function in the init script generates a similar but not identical version.

- **Tests are purely structural, not behavioral.** They verify that agent files contain required phrases (e.g., "NEVER read holdout") but cannot verify that Claude Code actually respects those constraints at runtime.

- **No CI/CD pipeline.** Tests must be run manually with `node --test tests/dark-factory-setup.test.js`.

- **The dark-factory/ subdirectory in this repo is mostly empty scaffolding.** It contains `manifest.json` (empty features object), a `README.md`, and `.gitkeep` files. In a real target project, this directory would accumulate specs, scenarios, and results during use.

## For New Features
- **Adding a new agent**: Create the `.md` file in `.claude/agents/` with proper frontmatter (name, description, tools). Add a corresponding generator function in `scripts/init-dark-factory.js`. Add the agent name to `expectedAgents` in the test file. Add information barrier tests if the agent has access restrictions.
- **Adding a new skill**: Create `SKILL.md` in `.claude/skills/{skill-name}/`. Add a corresponding entry in the `getSkillContent()` switch in `init-dark-factory.js`. Add the skill name to `expectedSkills` in the test file.
- **Modifying agent behavior**: Update BOTH the `.md` file and the generator function in `init-dark-factory.js`. Run tests to verify frontmatter and barrier constraints still pass.
- **Test expectations**: Every agent must have valid frontmatter. Every information barrier must be tested. Every new slash command must be documented in CLAUDE.md.

## For Bug Fixes
- **Running tests**: `node --test tests/dark-factory-setup.test.js` (all 113 tests, ~200ms)
- **Running a specific suite**: Not directly supported by the test structure; all tests are in a single file
- **Common failure patterns**: Frontmatter name mismatch (agent filename vs frontmatter `name` field), missing barrier phrases in agent content, init script generating content that doesn't match the source `.md` files
- **Fragile areas**: The init script's template literals are the most fragile part — escaped backticks, nested template strings, and the dual-source-of-truth problem between `.md` files and generator functions
