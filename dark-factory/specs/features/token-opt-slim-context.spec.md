# Feature: token-opt-slim-context

## Context

Dark Factory agents load `project-profile.md` and `code-map.md` as full-context documents before doing any work. As projects grow, these files grow too — the current Dark Factory project-profile runs ~155 lines (~2 200 tokens) and code-map runs ~135 lines (~2 000 tokens). For the architect-agent in particular, which spawns 3 instances in parallel, the combined context load per pipeline run is expensive.

Most agent roles do not need the full document. An architect reviewing a spec needs: the tech stack summary, the critical conventions it must not violate, the top entry points for blast radius, the gotchas that will bite the implementation, and the highest-fan-in hotspots that a change must not break. Everything else is rarely consulted.

This spec introduces pre-generated slim variants — `project-profile-slim.md` (~500 tokens) and `code-map-slim.md` (~400 tokens) — as first-class artifacts generated alongside the full files. The slim files are permanent, committed artifacts written by the same agents that write the full files: `onboard-agent` writes `project-profile-slim.md` and `codemap-agent` writes `code-map-slim.md`. `df-cleanup` refreshes both whenever it runs its health check so the slim files never silently lag behind the full files.

**Who reads the slim files is handled in a separate spec (`token-opt-architect-review`).** This spec covers generation only.

## Scope

### In Scope (this spec)

- `onboard-agent.md` gains a new phase (after writing `project-profile.md`) that writes `dark-factory/project-profile-slim.md` from the full profile using a defined extraction rule.
- `codemap-agent.md` gains an instruction to write `dark-factory/code-map-slim.md` as a second output alongside `dark-factory/code-map.md`, carrying the same `Git hash:` header.
- `dark-factory/templates/project-profile-slim-template.md` — new template defining the structure and section order for the slim profile.
- `df-cleanup/SKILL.md` gains a slim-file refresh step: whenever it runs health checks, re-generate both slim files from the current full files.
- Plugin mirrors for all four files above (`plugins/dark-factory/agents/onboard-agent.md`, `plugins/dark-factory/agents/codemap-agent.md`, `plugins/dark-factory/templates/project-profile-slim-template.md`, `plugins/dark-factory/skills/df-cleanup/SKILL.md`).
- Tests in `tests/dark-factory-setup.test.js` verifying: slim files are generated by the right agents, contain required sections, and carry the `Git hash:` header (codemap-slim only).

### Out of Scope (explicitly deferred)

- **Architect-agent loading behavior.** Which agent reads which slim file, and under what conditions, is handled in `token-opt-architect-review`. This spec does not touch `architect-agent.md`.
- **`implementation-agent.md`, `spec-agent.md`, `code-agent.md`, `debug-agent.md`.** Not in scope.
- **`spec-template.md`, `debug-report-template.md`.** Not in scope.
- **Automatic staleness recovery during pipeline runs.** If a slim file is missing mid-pipeline, that is a consumer concern handled in `token-opt-architect-review`. This spec specifies the generation trigger only.
- **Token measurement CI gate.** The 500/400-token targets are soft targets — agents apply judgment. No automated gate is in scope.
- **On-demand slim generation by any other agent.** Only onboard-agent, codemap-agent, and df-cleanup generate slim files. Other agents read them (handled in the consumer spec).

### Scaling Path

If additional agents need slimmed context for other files (e.g., a `manifest-slim.md` or per-module summaries), the same extraction pattern (full-file → template-driven subset → permanent artifact, same-hash header) is repeatable. The consumer spec (`token-opt-architect-review`) can be expanded to load additional slim files without modifying this spec's generation logic.

## Requirements

### Functional

- FR-1: `onboard-agent.md` MUST include a new phase — after writing `project-profile.md` and before spawning any other agent — that generates and writes `dark-factory/project-profile-slim.md`. — Ensures slim profile is always co-generated with the full profile.
- FR-2: `project-profile-slim.md` MUST be generated from the full `project-profile.md` by extracting exactly: (a) a header disclaimer line, (b) the Tech Stack table (rows only, no prose), (c) 3–5 critical-conventions bullet points from the Architecture / Structural Notes sections, (d) the top 2–3 entry points from Entry Point Traces, (e) the Common Gotchas section verbatim. — These five elements are the highest-value sections for any reviewer.
- FR-3: `project-profile-slim.md` MUST begin with a single-line header: `> Slim profile — generated from project-profile.md. For full context read project-profile.md.` — Consumers need to know the file is derived, not authoritative.
- FR-4: The target size of `project-profile-slim.md` is ~30 lines / ~500 tokens. — Agents applying this spec may exceed the target for genuinely complex projects; the target is a guide, not a hard cap.
- FR-5: `codemap-agent.md` MUST be updated so that `dark-factory/code-map-slim.md` is listed as a second required output alongside `dark-factory/code-map.md`. Both files MUST be written in the same operation (full-scan or incremental refresh). — Atomic co-generation ensures the slim file is never missing when the full file exists.
- FR-6: `code-map-slim.md` MUST contain exactly: (a) a header disclaimer line, (b) the Shared Dependency Hotspots table verbatim, (c) the Module Dependency Graph section — names + arrow targets only (no prose descriptions). — These two sections provide the blast-radius signal needed for architecture review.
- FR-7: `code-map-slim.md` MUST begin with: `> Slim map — generated from code-map.md. For full context read code-map.md.` — Same rationale as FR-3.
- FR-8: `code-map-slim.md` MUST carry the SAME `Git hash:` header line as `code-map.md` — identical hash value, same header format. — The git hash is the existing freshness signal used by df-intake, df-debug, and df-orchestrate. Slim files participate in this protocol so any freshness check against code-map.md also applies to code-map-slim.md.
- FR-9: The target size of `code-map-slim.md` is ~25 lines / ~400 tokens. — Same guidance as FR-4.
- FR-10: `code-map-slim.md` MUST NOT include: Entry Point Traces, Interface/Contract Boundaries, Cross-Cutting Concerns, Circular Dependencies, or Dynamic/Runtime Dependencies sections. — These sections are valuable for code-agents and debug-agents but are not the primary concern during architect review for blast radius.
- FR-11: When `codemap-agent` runs in **incremental refresh mode**, it MUST update `code-map-slim.md` in the same refresh pass as `code-map.md`. — Ensures slim file does not fall behind after a stale-map refresh triggered by df-intake/df-debug/df-orchestrate.
- FR-12: A new template `dark-factory/templates/project-profile-slim-template.md` MUST be created defining: the header disclaimer, section names, and the extraction rule for each section (which source section feeds which slim section). — Onboard-agent uses this template as the structure reference when generating the slim profile.
- FR-13: `df-cleanup/SKILL.md` MUST gain a slim-file refresh step executed whenever df-cleanup runs its health check. The step MUST: (a) check whether `dark-factory/project-profile.md` exists — if yes, re-generate `project-profile-slim.md` from it; (b) check whether `dark-factory/code-map.md` exists — if yes, re-generate `code-map-slim.md` from it (preserving the git hash header from the full file). — Ensures slim files are refreshed even if they were generated by an older version of the extraction logic.
- FR-14: The df-cleanup slim refresh step MUST run before the promoted-test health check (Step 2 in the existing skill), so that agents spawned from cleanup see fresh slim files. — Ordering matters; a stale slim file loaded mid-cleanup is a silent error.
- FR-15: Every new or changed file under `.claude/` or `dark-factory/templates/` MUST have an exact mirror in `plugins/dark-factory/`. — Plugin mirror contract is enforced by existing contract tests; this spec must not break them.
- FR-16: `tests/dark-factory-setup.test.js` MUST contain assertions verifying: (a) onboard-agent generates `project-profile-slim.md`, (b) codemap-agent generates `code-map-slim.md`, (c) codemap-agent instructions state the slim map carries the same git hash as the full map, (d) `code-map-slim.md` excludes the out-of-scope sections (FR-10), (e) `project-profile-slim.md` includes the required sections (FR-2), (f) the slim template file exists and contains section definitions for all required elements. — Locks the generation contract so implementation cannot silently omit a required section.

### Non-Functional

- NFR-1: Slim file generation MUST NOT require an additional developer confirmation step — it is automatic, a sub-operation of the full-file write. — Developer sign-off already happens on the full profile/map; requiring a second confirmation for the slim variant adds friction with no quality benefit.
- NFR-2: Slim files MUST be valid markdown that renders cleanly in standard markdown viewers. — Slim files will be read by agents and potentially inspected by developers in GitHub.
- NFR-3: Both slim files MUST be committed to git alongside their parent full files. They are permanent artifacts, not ephemeral cache. — Ephemeral slim files would break any workflow that starts from a git checkout without running onboard or codemap first.
- NFR-4: The extraction logic described in FR-2 and FR-6 MUST be expressed in natural-language instructions inside the agent files — not as code. — Dark Factory is a prompt-engineering framework; all logic is expressed in markdown agent definitions.

## Data Model

No runtime data model changes. This spec introduces two new permanent artifact files:

### `dark-factory/project-profile-slim.md`
```
> Slim profile — generated from project-profile.md. For full context read project-profile.md.

## Tech Stack
| Layer | Technology |
|-------|-----------|
{rows from full profile Tech Stack table — verbatim}

## Critical Conventions
- {convention 1} — sourced from Architecture/Patterns to Follow or Structural Notes
- {convention 2}
- ... (3–5 total)

## Entry Points
{top 2–3 entry points from full profile Entry Point Traces, names only}

## Common Gotchas
{verbatim copy of the Common Gotchas section from the full profile}
```

### `dark-factory/code-map-slim.md`
```
> Slim map — generated from code-map.md. For full context read code-map.md.
> Auto-generated by Dark Factory codemap-agent. Last analyzed: {ISO date}
> Git hash: {identical to code-map.md}
> Coverage: {identical to code-map.md}

## Shared Dependency Hotspots
{verbatim copy of Shared Dependency Hotspots table from code-map.md}

## Module Dependency Graph
{names + arrow targets from code-map.md Module Dependency Graph — omit inline prose descriptions and file-level details}
```

### `dark-factory/templates/project-profile-slim-template.md`

A new template file referenced by onboard-agent when generating the slim profile. Defines:
- Header disclaimer line (exact text)
- Section names and order: Tech Stack, Critical Conventions, Entry Points, Common Gotchas
- Extraction rule for each section (which source section to pull from and how)
- Target size guidance (30 lines / 500 tokens)

## Migration & Deployment

**Applies — but additive only.**

- **Existing data**: `project-profile-slim.md` and `code-map-slim.md` do not exist in any current installation. After this spec lands, a fresh `/df-onboard` run will generate them. Existing installations that have not re-run onboard will simply lack the slim files until: (a) they run `/df-onboard`, (b) they run `/df-cleanup`, or (c) any pre-pipeline hook triggers a codemap refresh. None of these is blocking — the consumer spec (`token-opt-architect-review`) handles the missing-slim-file fallback.
- **Rollback plan**: Revert the commit. The two slim files are additive artifacts; removing them breaks nothing (consumers fall back to full files, handled in the consumer spec). The agent/skill/template changes are additive only — no existing behavior is removed.
- **Zero-downtime**: Yes. There is no running service. Agents that run after this spec lands will generate slim files on their next onboard/codemap/cleanup run.
- **Deployment order**: Single wave. This spec is independently implementable. The consumer spec (`token-opt-architect-review`) MUST NOT land before this one — it depends on slim files existing.
- **Stale data**: If an older `code-map-slim.md` exists (from a previous version of the extraction logic), `df-cleanup` will regenerate it from the current full file on its next run. No special migration logic is needed.

## API Endpoints

N/A — this spec introduces no runtime APIs. All changes are agent instructions, a template, a skill update, and tests.

## Business Rules

- BR-1: **Slim files are derived, not authoritative.** The full files (`project-profile.md`, `code-map.md`) are the source of truth. Any consumer that needs a field not present in the slim file reads the full file. — Documented in the header disclaimer of each slim file.
- BR-2: **Slim files are co-generated with their full counterpart.** `project-profile-slim.md` is always written in the same operation as `project-profile.md`. `code-map-slim.md` is always written in the same operation as `code-map.md` (full scan or incremental refresh). — Prevents a state where the full file is fresh but the slim file is from a previous run.
- BR-3: **`code-map-slim.md` carries the same git hash as `code-map.md`.** The hash is not separately computed — it is copied from the full map header. — Existing freshness checks compare the stored hash in code-map.md to current HEAD. Slim files inherit the same staleness status automatically.
- BR-4: **`df-cleanup` always refreshes slim files.** Whenever df-cleanup runs, it regenerates both slim files from the current full files (if the full files exist). This is unconditional — it does not check whether the slim files are already fresh. — A cheap re-generation is safer than a staleness check that might have edge cases.
- BR-5: **Plugin mirrors must match source.** Every agent/skill/template file modified by this spec MUST have an identical copy in `plugins/dark-factory/`. — Existing convention; enforced by contract tests.
- BR-6: **Slim file generation is automatic and silent.** No developer confirmation prompt is shown for slim file generation. The prompt shown for the full file covers both. — Agents confirm the full file; slim is a mechanical derivative.

## Error Handling

| Scenario | Response | Side Effects |
|----------|----------|--------------|
| Full `project-profile.md` does not exist when df-cleanup tries to refresh slim profile | Skip profile-slim regeneration. Log: "project-profile.md not found — skipping slim profile refresh." Proceed with the rest of cleanup. | None |
| Full `code-map.md` does not exist when df-cleanup tries to refresh slim map | Skip code-map-slim regeneration. Log: "code-map.md not found — skipping slim map refresh." Proceed with cleanup. | None |
| Slim file write fails (disk full, permissions error) | Log the failure. Proceed with the rest of the operation (do not fail the full profile/map write). A missing slim file is tolerable; a missing full file is not. | Warning logged |
| Slim file exists but is missing its header disclaimer | Consumer reads it as-is. Staleness may not be surfaced. `df-cleanup` regenerates on next run, restoring the header. | None — self-healing |
| `code-map-slim.md` git hash differs from `code-map.md` git hash | This state should not occur (BR-3 and FR-11 prevent it). If detected by a test, it signals a bug in the codemap-agent extraction step. | Test fails |
| Plugin mirror out of sync after this spec's changes | Contract test fails. | Build breaks until mirror is synced |
| onboard-agent incremental refresh (re-run on existing project) | Re-generates `project-profile-slim.md` from the new full profile. Overwrites any previous slim file. | Previous slim overwritten — expected |

## Edge Cases

- EC-1: **Incremental refresh of codemap-agent** — when codemap-agent is triggered by a pre-pipeline hook (not onboard), it runs in refresh mode. `code-map-slim.md` must be refreshed in the same pass, not skipped because "it was generated earlier." Expected: slim map is updated atomically with the full map.
- EC-2: **`df-cleanup` runs before any onboard has ever been run** — neither full files nor slim files exist. Expected: df-cleanup skips the slim refresh steps gracefully and logs the skips. The rest of cleanup continues normally.
- EC-3: **`df-cleanup` runs after onboard but slim files were manually deleted** — the full files exist but the slim files do not. Expected: df-cleanup regenerates both slim files from the full files.
- EC-4: **Incremental refresh truncated (Coverage: PARTIAL)** — codemap-agent's refresh set is capped at 20 modules. The slim map is regenerated from the partial full map. Expected: slim map reflects whatever is in the full map; the `Coverage: PARTIAL` flag is copied into the slim map header.
- EC-5: **Very small project** — the full profile/map is shorter than the slim target. Expected: the slim file is essentially the same as the full file (no sections to omit), still carrying the header disclaimer and the same `Git hash:` for code-map-slim. No error.
- EC-6: **Full profile exists but has no Common Gotchas section** (e.g., greenfield project with minimal profile) — Expected: the onboard-agent omits the Common Gotchas section from the slim profile rather than writing an empty section. The slim profile is still valid.
- EC-7: **Full profile exists but has no Entry Point Traces** — Expected: same treatment as EC-6 — section is omitted, not written empty.
- EC-8: **Plugin mirror drift** — developer edits onboard-agent.md but forgets to update the plugin mirror. Expected: contract test fails, blocking the landing.
- EC-9: **onboard-agent incremental refresh (re-run)** — developer re-runs `/df-onboard` on a project that already has slim files. Expected: slim files are overwritten with the new extraction from the refreshed full profile/map. No confirmation prompt for the slim files specifically.
- EC-10: **code-map-slim.md git hash does not match code-map.md git hash** — indicates a bug in the generation step. Expected: this state should not exist; test suite detects and fails if it verifies the hash parity claim (content-level assertion against the agent instruction, not a runtime check).

**None — this spec is independently implementable.**

- **Depends on**: no other sub-spec. The full files (`project-profile.md`, `code-map.md`) already exist; this spec adds the slim derivatives.
- **Depended on by**: `token-opt-architect-review`. That spec wires architect-agent to load slim files by default; it requires this spec to have landed so slim files exist in the repo.

## Implementation Notes

- **Agent file pattern**: Both agent files use markdown with YAML frontmatter. Add the slim-generation step as a numbered phase in the agent's process, matching the style of existing phases (e.g., Phase 3.5 in onboard-agent). Do not add a new heading-level section — add a phase within the existing Constraints block or the appropriate process section.
- **Slim phase placement in onboard-agent**: Insert after "Write `dark-factory/project-profile.md`" (Phase 7) and before Phase 7.5 (Git Hook Setup). Name it "Phase 7.2: Generate Slim Profile". Use language consistent with the agent: "Generate and write `dark-factory/project-profile-slim.md`..."
- **codemap-agent output block**: The agent's template block already lists two output files. Extend it to three. Add the slim-generation instruction to both the full-scan path (Step 3) and the incremental refresh path. Keep the sign-off note: slim file is always written without a separate sign-off — it is a mechanical derivative.
- **df-cleanup step insertion**: Insert before Step 2 (Promoted Test Health Check). Name it "Step 1.5: Slim File Refresh". The step reads the full files and regenerates slim files. Keep it short — 3–5 bullet points. Log each action (generated / skipped).
- **Template file format**: Follow `dark-factory/templates/project-profile-template.md` as the style model — use a `> Auto-generated by...` header note, then section headings with extraction rules in `{curly braces}` or inline prose.
- **Test conventions**: Tests use `node:test` + `node:assert/strict`, no external deps. New tests should follow the `describe`/`it` block pattern used in the rest of `dark-factory-setup.test.js`. Assertions should be string-match assertions against agent/skill file content — e.g., `assert.ok(onboardAgent.includes("project-profile-slim.md"), ...)`.
- **Plugin mirror**: Apply identical changes to plugin mirrors. Contract tests (`dark-factory-contracts.test.js`) compare source and plugin content byte-for-byte. Run tests locally before considering the implementation complete.
- **Do NOT add a developer confirmation prompt** for the slim files. The slim generation is silent — no AskUserQuestion call, no pause.
