## Domain Review: Security & Data Integrity

### Feature: playwright-test-hardening
### Reviewer: architect-agent (Security & Data Integrity domain)
### Status: APPROVED WITH NOTES

---

### Findings

#### Blockers

None.

#### Concerns

**C-1: Dev server command injection via project profile field (Medium)**
FR-2 instructs the test-agent to read a "Dev Server" field from the project profile and execute it as a shell command. The project profile is auto-generated by the onboard-agent, but it is a markdown file that any developer (or agent) can edit. A malicious or accidental value in the Dev Server field (e.g., `rm -rf / && npm run dev`) would be executed verbatim by the test-agent.

**Mitigation already partially present**: The test-agent runs inside Claude Code's sandboxed tool execution, which requires user approval for shell commands. However, the spec should explicitly note that the dev server command is executed via the Bash tool (subject to permission checks) and that agents must never construct shell commands by interpolating untrusted strings without the Bash tool's permission gate.

**Recommendation**: Add a note to the Implementation Notes for Step 2.5 stating: "The dev server command from the project profile is passed to the Bash tool as-is. Claude Code's permission system gates execution. Do not wrap it in `sh -c` or concatenate it with other commands."

**C-2: Port assumption on "address already in use" (Low)**
EC-11 and the error handling table specify that if the dev server port is already in use, the test-agent should assume the server is already running and proceed. This is generally safe, but in a shared development environment, the port could be occupied by an unrelated process. Tests would then run against the wrong server, potentially producing misleading results (not a data integrity issue per se, but a correctness concern that borders on integrity).

**Recommendation**: The spec already handles this reasonably. No change required, but worth documenting as a known limitation.

#### Suggestions

**S-1: Sanitize UI Layer field before comparison**
FR-1 specifies case-insensitive matching on the UI Layer field. The spec correctly notes trimming (EC-2: whitespace treated as missing). Consider also stripping surrounding quotes — a profile field like `"none"` (with literal quotes from a YAML value) should still match. This is a minor sanitization edge case.

**S-2: Dev server health check should not follow redirects to external hosts**
Step 2.5 polls the expected port with HTTP GET. If the dev server responds with a 3xx redirect to an external URL, the health check could leak information about the test environment to that URL (the request itself reveals the IP and timing). This is extremely low risk in practice since dev servers redirect locally, but the spec could note that the health check should only verify a TCP connection or HTTP response on localhost, not follow redirects.

**S-3: Process cleanup on agent crash (already addressed)**
NFR-3 correctly mandates server cleanup even on crash. The spec handles this well. No additional action needed — noting it as a positive finding.

### Key Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | No blockers for security domain | This feature modifies agent markdown definitions, not application code. No auth systems, databases, migrations, or API surfaces are affected. The security surface is limited to shell command execution within Claude Code's permission model. |
| 2 | Dev server command execution is acceptable | The Bash tool permission gate provides the security boundary. The spec does not bypass or circumvent this gate. |
| 3 | Port-in-use assumption is acceptable | The "assume server is running" behavior on port conflict is pragmatic and does not create a data integrity risk. Worst case is tests running against the wrong server, which produces test failures (not data corruption). |
| 4 | No concurrent write concerns | All changes are to markdown definition files read at agent spawn time. No shared mutable state, no race conditions, no concurrent write scenarios. |
| 5 | No data exposure concerns | Agent definitions do not contain secrets, credentials, or sensitive data. The flakiness reporting format (`flaky-e2e`, `flakyE2E`) does not leak implementation details beyond test status. |
